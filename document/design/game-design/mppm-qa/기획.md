# mppm-qa — 멀티 QA 검증 기준 기획서

> 슬러그: `mppm-qa` · 작성: director · 2026-06-02
> 본 문서는 **무엇을 멀티에서 검증해야 하는가** = 검증 시나리오 기준만 정의한다.
> 코드 방법(부트스트랩·프로브)은 client, 검증 절차·리포트는 qa 영역.

---

## 1. 한줄 컨셉

호스트와 클론 양쪽의 게임 상태를 대조해, "한쪽에서 일어난 일이 반대쪽에 정확히·제때 반영됐는가"를 사용자 개입 0으로 자동 판정하는 멀티 동기화 검증 기준.

## 2. 디자인 골 (Why)

- WES는 Host=Server 권위 구조다. **권위측(호스트)에서 일어난 상태 변화가 클라이언트(클론)에 누락·지연·왜곡 없이 전파**되는 것이 멀티 게임의 생명선이다.
- 1인 개발에서 멀티 동기화 버그는 손으로 잡기 가장 비싸다 (창 2개 띄우고 눈으로 비교). 이를 **기계가 양측 스냅샷을 대조**해 자동 검출한다.
- 검증 기준이 명확해야 qa가 "통과/실패"를 일관되게 판정할 수 있다. 본 문서가 그 판정의 기준선이다.

## 3. 안티 골 (What this is NOT)

- **게임플레이 재미 검증이 아니다.** "전투가 재미있는가"는 다루지 않는다. 오직 동기화 정확성.
- **싱글 플레이 로직 검증이 아니다.** 호스트 단독에서 HP가 깎이는지는 기존 싱글 QA(모드 A/B) 영역. 여기는 **양측 대조**만.
- **성능/프레임/네트워크 대역폭 측정이 아니다.** 동기화의 "정확성"이지 "효율"이 아니다. (지연 시간은 성공 기준의 상한선으로만 사용)
- **4인 풀 부하 테스트가 아니다.** 1차 목표는 host + 클론 1 (2인) 대조. 4인은 백로그.

## 4. 플레이어 경험 흐름 (= QA 실행 흐름)

본 주제에는 "플레이어"가 없다. QA 에이전트가 사용자다. 그 경험 흐름:

```
QA 명령 1회
  → host + 클론 자동 접속, 양쪽 Ingame 진입
  → 시나리오별 호스트 권위 동작 자동 실행
  → 동작 직후 양측 스냅샷 수집
  → 기준표(아래 6장)와 대조 → 항목별 PASS/FAIL
  → 정리 후 리포트
  (전 과정 클릭 0)
```

## 5. 핵심 디자인 결정

### 5-1. 검증의 단위 = "동기화 명제(assertion)"
각 검증은 하나의 명제로 환원된다. 예: *"호스트가 몬스터를 스폰하면, 클론의 월드에도 동일 ID 몬스터가 동일 위치(±오차)로 존재한다."*
- 명제는 **항상 양측 스냅샷 비교**로 판정한다. 한쪽만 보면 통과로 치지 않는다.

### 5-2. 권위 방향을 명시한다
WES는 호스트 권위다. 검증 명제마다 **트리거 주체**를 명시:
- **호스트 트리거**: 몬스터 스폰/사망, 데미지 적용, NetworkVariable 갱신 → 클론에 전파 확인.
- **클론 트리거**: 클론이 입력(아이템 수집 요청, 이동) → ServerRpc로 호스트에 도달 → 호스트가 처리 → 결과가 다시 양측에 반영 확인. (왕복 검증)

### 5-3. 비교는 "값 일치"와 "존재 일치" 두 축
- **존재 일치**: 객체가 양측에 있는가/없는가 (스폰·디스폰·사망).
- **값 일치**: 양측 동일 객체의 상태값이 같은가 (HP, Cold, 위치, 인벤토리 수량).

### 5-4. 타이밍 허용 오차를 기준에 박는다
네트워크는 즉시가 아니다. 각 명제에 **수렴 허용 시간**을 둔다 (아래 8장). 그 시간 내 일치하면 PASS, 초과 후에도 불일치면 FAIL.

## 6. 검증해야 할 동작 목록 (콘텐츠 분량)

우선순위 P0(필수, 1차 구현) / P1(중요, 2차) / P2(백로그).

| # | 검증 명제 | 트리거 | 비교축 | 우선 |
|---|-----------|--------|--------|------|
| V1 | host·클론 양쪽 모두에서 **자기+상대 플레이어 캐릭터가 스폰**되어 존재한다 | 접속 | 존재 | P0 |
| V2 | 한 플레이어가 이동하면 **반대쪽 화면의 해당 캐릭터 위치**가 따라온다 (±위치 오차) | 클론/호스트 이동 | 값(위치) | P0 |
| V3 | 호스트가 **몬스터를 스폰**하면 클론에도 동일 몬스터가 존재한다 | 호스트 | 존재 | P0 |
| V4 | 데미지 적용 시 **몬스터/플레이어 HP(NetworkVariable)**가 양측 동일하다 | 호스트 | 값(HP) | P0 |
| V5 | **몬스터 사망** 시 양측에서 함께 사라지고, 드롭 아이템이 양측에 스폰된다 | 호스트 | 존재 | P0 |
| V6 | 클론이 **아이템을 수집**(ServerRpc)하면, (1) 클론 로컬 인벤 +1 [값, 클론 단측] + (2) 월드 드롭아이템이 **양측에서 디스폰**된다 [존재, 양측 대조] | 클론 | 값(클론단측)+존재(양측) | P0 |
| V7 | **Cold(NetworkVariable)** 변화가 양측 동일하다 | 호스트 | 값(Cold) | P0 |
| V8 | 클론이 **건물을 스폰**하면 양측 월드에 동일 건물이 존재하고, 클론 인벤토리에서 재료가 차감된다 | 클론 | 값+존재 | P1 |
| | ⚠ P1 진입 시 결정: `BuildingInfoId`가 plain int(복제 안 됨)라 클론에서 건물 **종류** 대조 불가. (a)종류 NetworkVariable 승격 vs (b)종류 대조 빼고 존재 대조만. → 기본 권고 **(b)**: V8 핵심은 "건물이 양측에 스폰됐나"(존재). 종류 대조는 건물 다양성이 실제 멀티 버그원이 될 때 (a)로 승격. P0 영향 0. | | | |
| V9 | **늦은 조인(late join)**: 호스트가 이미 진행 중인 상태(스폰된 몬스터·건물·HP 변화)에 클론이 뒤늦게 접속해도, 클론이 현재 월드 상태를 정확히 받는다 | 접속 순서 | 존재+값 | P1 |
| V10 | **디스커넥트**: 클론이 나가면 호스트 월드에서 해당 플레이어가 정리되고, 호스트는 게임을 계속 유지한다 | 클론 이탈 | 존재 | P1 |
| V11 | 4인 동시 접속 시 위 명제들이 모두 유지된다 | 다중 접속 | 전체 | P2 |
| V12 | **재접속**: 나갔던 클론이 다시 들어와도 V9가 성립한다 | 재접속 | 존재+값 | P2 |

**1차(P0) 분량**: 검증 명제 **7개 (V1~V7)**. 이것이 멀티의 최소 생존선. (V7 Cold는 HP와 동일 메커니즘·추가 비용 0이라 P1→P0 승격 — client 실측 확정 2026-06-02.)

## 7. 톤 가이드

이 주제는 게임 콘텐츠가 아니므로 게임 톤(Don't Starve 어두움)은 적용 대상이 아니다.
대신 **검증 리포트의 톤** 기준:
- 판정은 **이진(PASS/FAIL)** + 불일치 시 **양측 실제값 병기**. ("FAIL: host HP=80, clone HP=100")
- 모호한 "아마 동기화됨" 금지. 수렴 시간 내 일치 못 하면 FAIL.

## 8. 밸런스/튜닝 포인트 (= 성공 기준 임계값)

| 항목 | 기준값 | 비고 |
|------|--------|------|
| 위치 동기화 허용 오차 | ≤ 0.5 유닛 | V2 위치 비교. **이동 중 순간이 아니라 "정지 후 수렴값"으로 비교** (보간 지연 오판 방지). client 실측: ClientNetworkTransform 오너권위+보간 |
| 스폰/디스폰 수렴 시간 | ≤ 1.0초 | V1·V3·V5·V8. 이 시간 내 양측 존재 일치 |
| NetworkVariable 수렴 시간 | ≤ 0.5초 | V4·V7. tick rate 고려 |
| RPC 왕복 반영 시간 | ≤ 1.0초 | V6·V8 클론 트리거 왕복 |
| HP/Cold 값 일치 허용 오차 | 0 (정확 일치) | NetworkVariable<int> Read=Everyone → 클론도 권위측 정확값. client 실측으로 오차 0 확인 |
| late join 상태 수렴 시간 | ≤ 2.0초 | V9. 초기 스냅샷 전송 여유 |

> client 실측 확정(2026-06-02): TickRate=30, 로컬 직결(127.0.0.1) 채택 → 위 임계값 전부 여유, 조정 불필요.

### V6 인벤토리 모델 주의 (client 실측 반영)
인벤토리는 NetworkVariable/NetworkList가 **아니다** — 아이템 수집은 수집자에게만 ClientRpc(Single)로 로컬 인벤 반영("각자 로컬, 본인 것만", 설계상 정상). 따라서 V6의 "양측 인벤 대조"는 성립 불가 → **양측 대조 축은 월드 드롭아이템 디스폰(존재)**에, 인벤 +1은 클론 단측 값 검증으로 분리한다. (향후 거래/공유창고 콘텐츠 도입 시 복제 모델 재검토 — 10장 백로그)

## 9. 엣지 케이스 (게임플레이/시나리오 차원)

- **접속 순서**: 호스트 먼저 → 클론 (정상). 클론이 호스트보다 먼저 Ingame 진입 시도 시 처리 (V1 전제 검증).
- **동시 트리거 경합 (P1, V6-경합)**: 두 플레이어가 같은 월드 아이템을 동시에 수집 시도 → 호스트 권위로 1명만 성공, 양측에 일관 반영. P0 V6 본체(단독 수집)와 분리 — P0는 "전파가 되는가", 경합은 "충돌을 권위가 옳게 푸는가"로 검증 목적이 다르고 P0 통과가 선행돼야 의미 있다. P0 6명제는 단순·단독 트리거로 유지.
- **사망 중 상호작용**: 죽은 몬스터에 추가 데미지 RPC가 와도 양측 일관 (음수 HP·중복 드롭 없음).
- **빠른 연속 동작**: 짧은 시간에 여러 스폰/데미지가 쏟아질 때 순서·누락 없이 양측 일치.
- **late join 타이밍**: 호스트가 몬스터를 스폰하는 *바로 그 순간* 클론이 조인 (V9 경계).

## 10. 향후 확장 (Backlog)

- V11(4인), V12(재접속) 정식 편입.
- 동기화 지연 시간 통계(평균/최대) 리포트화 — 정확성 넘어 품질 지표.
- 회귀 스위트: P0 6명제를 커밋마다 자동 회귀로.
- 게임 신규 콘텐츠 추가 시(예: 신규 몬스터 AI, 신규 상호작용) 검증 명제 자동 확장 가이드.
- **인벤토리 복제 모델 재검토**: 현재 인벤은 로컬 단측. 거래/공유창고/드롭 가시성 같은 "타인 인벤을 봐야 하는" 콘텐츠 도입 시 NetworkList 등 복제 모델 필요 → 그때 V6를 양측 인벤 대조로 승격.
- Relay 경로 자체의 연결 검증 (현재 로컬 직결 채택, Relay는 P2 백로그).
