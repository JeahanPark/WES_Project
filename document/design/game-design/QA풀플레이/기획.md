# QA 풀플레이(E2E 자동 진행) — 기획서

> **작성일**: 2026-05-31
> **유형**: 개발/QA 편의 도구 (플레이어 비노출)
> **상태**: 초안 — client 검토 대기

---

## 1. 한줄 컨셉

> 한 번 트리거하면 사람 입력 없이 **Intro → Lobby → Ingame → 탈출 클리어**까지 게임 루프 전체를 자동으로 통과하는 회귀/E2E 테스트 모드.

---

## 2. 디자인 골 (Why)

- **회귀 안전망**: 코어 흐름(씬 전환·Host 셋업·스폰·이동·탈출 RPC·클리어/로비 복귀)이 깨졌는지 사람 손 없이 1번에 감지.
- **재현성**: 매번 동일 경로·동일 조건으로 돌아 "어제는 됐는데 오늘 막힘"을 빠르게 짚는다.
- **사용자 요구 충족**: 시작점은 **Intro 씬**. Ingame 단독이 아니라 진입 흐름 자체도 검증 대상.

## 3. 안티 골 (What this is NOT)

- 플레이어용 콘텐츠/튜토리얼/데모 모드가 **아니다**. 출시 빌드 노출 금지.
- 전투 AI·자원 수집·제작의 *내용*을 검증하는 도구가 **아니다**. (그건 별도 단위 테스트 영역)
- 멀티플레이 동기화 부하 테스트가 **아니다**. 로컬 Host 1인 기준.
- 기존 `m_TestMode`(Ingame 단독 진입)를 대체/삭제하지 **않는다**. 공존·재사용한다.

---

## 4. 자동 진행 흐름 (E2E 시나리오)

```
[트리거]
   ↓
Intro 씬: START 자동 진입 (UI 클릭 또는 흐름 스킵)
   ↓
Lobby 씬: 방 생성 → Host 시작 → START 자동
   ↓
Ingame 씬: 로컬 플레이어 스폰 확인
   ↓
로컬 플레이어 → EscapePoint 까지 자동 이동
   ↓
탈출 RPC 발동 → 클리어 화면
   ↓
[판정] 클리어 화면 도달 = PASS / 타임아웃 = FAIL
   ↓
로비 복귀까지 확인 후 종료
```

---

## 5. 핵심 디자인 결정 (4개)

### 결정 1 — 진입 범위: **Intro 시작 → 씬 진입점 직접 호출로 풀 흐름 통과** [확정]
- **Intro 씬에서 시작**(사용자 요구 충족). TestManager 코루틴이 UI 클릭이 아니라 각 씬 진입점을 직접 호출해 Ingame까지 진행:
  - LoginPopup 상당 → `LoadScene(Lobby)` → `HostRelayAsync` await → `LoadScene(Ingame)`(호스트 SceneManager 경로) → 스폰 대기.
- client 사실: Lobby Host는 Relay 비동기라 클릭이든 직접호출이든 같은 대기를 타므로, 클릭 시뮬레이션은 추가 위험만 늘림 → 진입점 직접 호출이 견고.
- 결과: Intro·Lobby·Ingame **씬전환·Host셋업·스폰 회귀까지 실제 경로로 커버**. (Lobby UI 위젯 클릭 회귀만 제외 — Backlog)
- **양보 불가(준수됨)**: 시작점 Intro 씬.

### 결정 2 — 클리어 경로: **경로탐색 실제 이동(MoveTo) + Warp 폴백** [확정]
- client 사실: PlayerCharacter는 SetDestination을 안 씀(매 프레임 direction Move). "SetDestination 한 방"은 작동 안 함.
- 채택: **CharacterBase에 범용 이동 API `MoveTo(Vector3)` 신규**(NavMesh 경로탐색으로 corner 산출 → 매 프레임 다음 corner 방향 MoveWithDirection). 기존 이동/애니/경계차단 전부 재사용 → **테스트 전용 분기 아님**(결정 4 원칙 부합).
- 이 MoveTo로 **EscapePoint까지 실제 걸어서 도달**(지형 우회 포함). "이동→트리거 발동" 사슬 실제 검증이 핵심 가치.
- **NavMesh 경로 없음/이동 타임아웃 시 → Warp 폴백** 후 트리거 진입. 폴백 발동 시 "이동 실패→Warp 폴백" 로그 1줄(회귀 신호).
- **거부**: OnPlayerReachedEscape 직접 호출(이동 스킵), Warp 단독, 직선 추적(나무/슬로프 끼임으로 의도 약화).
- 무적/스탯 동결: 기존 `SetMaxHP/SetHP(huge)` + `SetCold(0)` 관행 재사용(신규 무적 플래그 안 씀).

### 결정 3 — 실패 처리: **타임아웃 + 단계별 로그**
- 전체 시나리오 상한 타임아웃 **120초**. 초과 시 FAIL 판정 후 중단.
- 단계별 체크포인트 5줄: `Intro 자동시작` / `Ingame 진입·스폰 OK` / `이동 개시` / `EscapePoint 도달 OK (or Warp 폴백)` / `클리어 화면 OK`.
- 이동 단계 타임아웃 **60초** → Warp 폴백 트리거(결정 2).
- FAIL이어도 콘솔에 어느 단계에서 멈췄는지 명확히 남긴다.

### 결정 4 — 구조 수정 권한 / 트리거 [확정]
- 트리거: **TestManager 신규 코루틴 `TestFullPlayClear()` 단일 진입**. QA가 `u_play`로 Intro 씬 진입 후 `u_play_invoke`로 호출. 단축키/메뉴/부트 플래그 안 씀(트리거 1회 명확, 빌드 오염 없음).
- 진입 흐름·클리어 RPC에 **구조 문제**가 있으면 client와 수정 범위 합의 후 진행, team-lead에 보고.
- 원칙: **테스트 전용 별도 분기 최소화**. 기존 진입점/메서드 재사용 우선(MoveTo도 범용 API로 추가).

## 6. 콘텐츠 분량 (수치)

| 항목 | 값(제안) | 비고 |
|---|---|---|
| 자동화 단계 | 5단계 | Intro/Lobby/스폰/이동/클리어 |
| 전체 타임아웃 | 120초 | 초과 시 FAIL |
| 이동 단계 타임아웃 | 60초 | EscapePoint 미도달 시 FAIL |
| 체크포인트 로그 | 5줄 | 단계별 1줄 |
| 대상 플레이어 | 로컬 Host 1인 | 멀티 미포함 |

---

## 7. 톤 가이드

해당 없음 — 플레이어 비노출 개발 도구. 콘솔 로그는 사람이 빠르게 읽을 수 있게 단계명 + OK/FAIL 명확히.

---

## 8. 밸런스/튜닝 포인트

- 타임아웃 값(120초/60초)은 맵 크기·이동 속도에 따라 조정. 첫 실측 후 확정.
- 무적/스탯 동결 적용 시점(스폰 직후) 및 해제 시점(클리어 후 또는 모드 종료).

## 9. 엣지 케이스 (게임플레이/흐름 차원)

- **네트워크 연결 전제**: Host 셋업이 실 Relay(HostRelayAsync)에 의존 → 오프라인 환경에선 1단계(Host셋업)에서 FAIL. 이는 도구 결함이 아니라 결정1에서 의도적으로 받아들인 Relay 의존(실제 진입 경로 회귀 검증을 위한 트레이드오프). QA 실행 시 "네트워크 연결 전제" 명시.
- EscapePoint가 NavMesh로 도달 불가한 위치 → 이동 타임아웃으로 FAIL, 좌표/경로 로그.
- 클리어 화면 진입 후 로비 복귀 실패 → 별도 체크포인트로 분리해 어디서 끊겼는지 식별.
- 스폰 전 이동 명령 발동 → 스폰 완료 확인 후 이동 시작(순서 보장).

## 10. 향후 확장 (Backlog)

- 다중 경유지(자원 수집 1회 → 제작 1회 → 탈출) 시나리오로 커버리지 확대.
- 멀티플레이(2~4인) E2E.
- 클리어뿐 아니라 [전멸] 경로 자동 검증.

## 11. 참고/영감

- 기존 `InGameController.m_TestMode` / `CoInitializeTestMode` (Ingame 단독 자동 Host) — 본 모드의 Ingame 단계 셋업을 재사용할 자산.
- GDD 게임 흐름(Intro→Lobby→Ingame) 및 탈출 시스템(EscapePoint 도달) 정의 준수.
