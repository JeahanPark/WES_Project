## 0. 디렉터 3질문 답 (요약 — 본문 상세는 §5, §11)

| # | 질문 | 결정 | 비고 |
|---|---|---|---|
| 1 | 멀티 플레이어별(개인) 해금 구현 비용 수용 가능? | **가능 (primary 채택)** — 비용 **S~M** | 인벤토리가 이미 "클라 로컬 + sender 한정 RPC" 구조라 개인 단위가 오히려 자연스러움. 파티 공유 폴백은 불필요 |
| 2 | 도면 즉시해금형(슬롯 미점유, 줍는 즉시 소비+해금만 남김) 가능? | **가능 (primary 채택)** — 비용 **S** | `WorldDropItem.CollectServerRpc → AddItemClientRpc` 패턴을 그대로 본떠 "AddItem 대신 UnlockRecipe" 분기. 인벤토리 미경유 |
| 3 | CraftPopup 잠긴 레시피 노출+잠금 오버레이(재배치 없이) 가능? | **가능 (primary 채택)** — 비용 **S~M** | 목록은 이미 카테고리 전체를 `SetData`로 그림. 셀에 오버레이 1장 추가 + DetailPanel에 잠금 분기만. 목록 필터링/재배치 없음 |

**총평: 3질문 모두 primary 채택 가능. 폴백 불필요.** 총 구현 비용 **M** (신규 클래스 2개 + 기존 4개 소폭 수정).

## 1. 영향 범위 요약

| 레이어 | 영향 | 변경 종류 |
|---|---|---|
| 데이터(CSV) | `ItemInfo.csv`에 도면 아이템 3종 추가, `BlueprintInfo.csv` 신규(도면→레시피 매핑) | 추가 |
| 해금 상태 | 신규 `RecipeUnlockRegistry`(플레이어 로컬, 세션 한정) | 신규 |
| 제작 판정 | `CraftDetailPanel.CanCraft`에 잠금 게이트 추가 | 수정 |
| 제작 UI | `CraftScrollCell` 잠금 오버레이, `CraftDetailPanel` 잠금 안내, 해금 시 갱신 | 수정 |
| 아이템 획득 | `WorldDropItem`에 "도면 타입" 분기(인벤토리 미경유 해금) | 수정 |
| 알림 UI | 신규 도면 획득 토스트(절제된 1줄) | 신규 |
| 네트워크 | 신규 RPC 없음 — 기존 `WorldDropItem` Server/ClientRpc 패턴 재활용. 해금 상태는 클라 로컬 | 재활용 |
| 영구 진행 | 없음(세션 한정). 저장/직렬화 추가 안 함 | — |

핵심 사실: **인벤토리(`InventoryRegistry`)는 NetworkVariable이 아닌 클라이언트 로컬 데이터**다. `WorldDropItem.CollectServerRpc`는 서버에서 Despawn만 하고, 아이템 추가는 `AddItemClientRpc`로 **줍는 본인 클라에만** 적용된다. 따라서 해금도 "줍는 본인 클라에만" 적용하면 자동으로 개인 해금이 된다. 별도 owner NetworkVariable 동기화 불필요.

## 3. 신규 필요 클래스/스크립트

### 3.1 `RecipeUnlockRegistry` (신규, Registry — 클라 로컬 데이터)
- 위치: `Assets/Scripts/InGameObjectData/RecipeUnlockRegistry.cs`
- 역할: 현재 세션·현재 클라이언트의 **해금된 CraftId 집합** 관리. 세션 한정, 직렬화 없음.
- 가이드 시그니처:
  - `bool IsUnlocked(int _craftId)` — 기본 5종은 항상 true(아래 §4 기본해금 판정), 도면 3종은 집합 포함 여부
  - `void Unlock(int _craftId)` — 집합에 추가, 신규일 때만 `OnUnlockChanged?.Invoke(_craftId)` 발화
  - `void Clear()` — 세션 리셋(인벤토리 `Clear` 패턴 동일)
  - `event System.Action<int> OnUnlockChanged`
- 내부: `HashSet<int> m_UnlockedCraftIds`. 멤버 `m_` 접두사 준수.
- 기본해금 판정은 레지스트리가 `BlueprintInfo` 데이터를 참조해 "도면 필요 CraftId가 아니면 항상 해금"으로 처리(§4 참조).

### 3.2 `BlueprintToast` (신규, Component — 토스트 UI)
- 위치: `Assets/Scripts/UI/HUD/BlueprintToast.cs` (HUD 카테고리)
- 역할: "○○ 도면을 익혔다" 1줄 토스트. 페이드 인/표시/페이드 아웃 후 자동 비활성. 톤 §7·§11.1 준수(파티클·팡파레 금지).
- 가이드 시그니처:
  - `void ShowMessage(string _message)` — 외부 진입점
  - `private IEnumerator CoShow(string _message)` — 코루틴 `Co` 접두사 준수
- 슬롯: `[SerializeField] private CanvasGroup m_CanvasGroup; [SerializeField] private TMPro.TextMeshProUGUI m_MessageText;`
- 부착: `InGameHUDWorker` 프리팹 하위(디자이너 제공). 호출은 해금 이벤트 구독으로.

### 3.3 (자동생성) `BlueprintInfo` 변환 클래스
- 위치: `Assets/Scripts/Info/Generator/ConvertInfo/BlueprintInfo.cs` (CSV 변환으로 자동생성 — 직접 작성 금지)
- 필드: `int Id; int BlueprintItemId; int UnlockCraftId;`

> 신규 클래스 **2개**(Registry, Toast) + 자동생성 1개. 신규 Manager/Controller/Worker 없음.

컬럼: `Id.INT,BlueprintItemId.INT,UnlockCraftId.INT`

| Id | BlueprintItemId | UnlockCraftId |
|---|---|---|
| 1 | 401 | 5 (나무 방패) |
| 2 | 402 | 6 (철검) |
| 3 | 403 | 7 (가죽 갑옷) |

- **이 CSV가 "도면 필요 CraftId"의 단일 진실원**. `RecipeUnlockRegistry.IsUnlocked`는 이 매핑에 UnlockCraftId로 등장하지 않는 CraftId면 항상 해금으로 본다(= 기본 5종 자동 상시).

## 5. 네트워크 동기화 항목

| 항목 | 권한 | 동기화 방식 |
|---|---|---|
| 도면 줍기 요청 | 클라 → 서버 | **기존** `WorldDropItem.CollectServerRpc` 재활용(신규 RPC 없음) |
| 도면 오브젝트 Despawn | 서버 | 기존 `NetworkObject.Despawn()` |
| 해금 적용 | 줍는 본인 클라 단독 | **기존** `AddItemClientRpc`의 자매 분기 `UnlockBlueprintClientRpc(SendTo.SpecifiedInParams = sender)` |
| 해금 상태 보관 | 클라 로컬 | `RecipeUnlockRegistry`(NetworkVariable 아님) — 인벤토리와 동일 정책 |

**개인 해금이 공짜인 이유**: 인벤토리 자체가 NetworkVariable로 동기화되지 않고 줍는 본인 클라에만 `AddItemClientRpc`로 적용된다. 해금도 같은 타깃팅(sender 단독)을 쓰면 별도 owner-NetworkVariable·동기화 코드 없이 개인 단위가 보장된다.

> 신규 `NetworkVariable` 0개, 신규 `ServerRpc`/`ClientRpc` 0개. `WorldDropItem`에 ClientRpc 1개(`UnlockBlueprintClientRpc`)만 추가하거나, 기존 `AddItemClientRpc` 안에서 도면이면 해금·아니면 인벤토리 추가로 분기(권장: 분기 1개로 단순화).

## 6. 상태 머신

레시피 셀 상태(클라 표시 기준):

| 상태 | 의미 |
|---|---|
| Unlocked | 정상 컬러, 제작 가능(재료 충족 시) |
| Locked | 회색/실루엣 + 자물쇠 오버레이, 제작 불가 |

전이:
- Locked → Unlocked: 대응 도면 줍기 → `RecipeUnlockRegistry.Unlock(craftId)` (단방향)
- Unlocked → Locked: **세션 리셋(`Clear`)일 때만**. 세션 중 역전 없음(불가능 전이로 명시 차단)
- 기본 5종: 항상 Unlocked. Locked 진입 불가(불가능 전이)

## 11. 엣지 케이스 (구현 차원)

| 케이스 | 처리 |
|---|---|
| 같은 도면 두 번 주움 | `Unlock`이 HashSet 신규일 때만 이벤트 발화. 두 번째는 무효(토스트도 안 뜸). 게임적 무해(기획 §9) |
| 멀티: 동료가 같은 종류 도면 주움 | sender 단독 적용 → 각자 별도 해금. 자동 충족(기획 §9) |
| 도면 줍는 순간 디스커넥트 | 서버 Despawn은 됐으나 ClientRpc 미수신 가능 → 해금 누락. 세션 한정이라 영향 경미. 도면 오브젝트는 이미 Despawn(중복 획득 없음). 허용 |
| CraftPopup 열린 채 해금 발생 | `OnUnlockChanged` 구독으로 즉시 셀/디테일 갱신. 닫혀 있으면 다음 Open 시 `SelectCategory`에서 최신 상태 반영 |
| 잠긴 레시피 재료 충분 + 제작 시도 | `CanCraft`가 잠금 선검사로 false. 버튼 비활성+"도면 필요" 라벨. 재료 충분해도 차단(기획 §9) |
| 잠긴 셀 클릭 | 디테일 표시는 하되(보이게 유지), 재료 대신 "이 레시피는 도면이 필요합니다" 안내 + 제작 버튼 비활성(기획 §11.2) |
| 세션 재시작/사망 | `RecipeUnlockRegistry.Clear()` — 인벤토리 Clear와 동일 지점. 기본 5종만 남음 |
| 도면 아이템 ID인데 BlueprintInfo 매핑 없음(데이터 누락) | `GetBlueprintByItemId` null → 해금 스킵 + `GameDebug.LogWarning`. 인벤토리 추가도 안 함(도면은 슬롯 미점유 정책). 데이터 결함은 경고로 가시화 |
| `InGameController.Instance`/`ObjectDataWorker` null(씬 미초기화 타이밍) | 줍기·해금 가드 후 무시(기존 코드도 null 가드 다수) |
| 도면 스폰 분포 미구현 상태에서 테스트 | `TestManager`로 도면 아이템 강제 스폰(기존 드롭 스폰 경로 재활용)하여 해금·UI 검증 가능 |

## 12. 권고

1. **도면 월드 배치(원거리 가중)는 별도 슬라이스로 분리** 권고. 현재 영역별 아이템 스폰 분포 시스템이 부재(몬스터만 `InGameAreaWorker` 영역 스폰). 이번 슬라이스는 "해금 메커니즘 + 제작창 잠금 UI + 알림"으로 닫고, 배치 가중은 후속으로. (해금 코어는 배치 방식과 독립적이라 먼저 완성 가능)
2. **`WorldDropItem` 분기는 1곳으로 단순화** 권고: 기존 `AddItemClientRpc` 안에서 "BlueprintInfo 매핑 있으면 Unlock, 없으면 AddItem"로 분기. 별도 RPC 추가보다 콜사이트 1개라 위험 낮음.
3. **분류 진실원을 `BlueprintInfo.csv` 한 곳으로 고정** 권고: CraftInfo.cs는 자동생성·수정금지이므로 "잠금 대상" 플래그를 CraftInfo에 넣지 말고 BlueprintInfo 매핑으로만 판정. 기획 §5.1 변경 시 CSV만 수정.
4. **세션 리셋 호출 지점 일원화**: `RecipeUnlockRegistry.Clear()`는 반드시 인벤토리 `Clear`와 같은 지점에 묶어, 리셋 누락(영구진행 안티골 위반)을 방지.
