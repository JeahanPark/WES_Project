## 0. 실현성 게이트 결론 (최우선 — 패키지 코드 직접 조사 완료)

조사 대상: `Library/PackageCache/com.unity.multiplayer.playmode@…` (버전 **1.6.2** 확인).

| 질문 | 결론 | 근거 |
|------|------|------|
| 가상 플레이어를 **프로그램적으로** 활성/유지 가능한가? | **부분 가능 — 권장 안 함** | 활성 API(`UnityPlayer.Activate`)는 전부 `internal`. asmdef `InternalsVisibleTo` 화이트리스트에 우리 어셈블리 없음 → 직접 호출 불가. 리플렉션 강제는 가능하나 깨지기 쉬움. |
| per-run 개입 0 달성 가능한가? | **달성 가능 (최초 1회 셋업으로)** | `Activate`가 `m_PlayerStateJson.Active=true`를 `SystemDataStore.SavePlayerJson`으로 **디스크 영속화**. 한 번 활성한 가상 플레이어는 프로세스가 켜진 채 유지, 재진입 시 `if(Active) return`으로 재사용. → **최초 1회 MPPM 창에서 Player2 활성**만 수동, 이후 매 실행 클릭 0. 브리프 1순위 제약 충족. |
| 클론/호스트를 **런타임 코드**에서 구분 가능한가? | **가능 (public API)** | `Unity.Multiplayer.Playmode.CurrentPlayer` 는 **public**. `CurrentPlayer.IsMainEditor`, `CurrentPlayer.ReadOnlyTags()` 로 역할/태그 판정. McpBridge가 이미 같은 판정을 리플렉션으로 사용 중(`IsCloneEditor`). |

**채택 방향 (효율적인 쪽):**
- 가상 플레이어 활성화 = **최초 1회 수동** (MPPM 창 Player2 체크). 그 외 전 자동.
- 부트스트랩이 `CurrentPlayer.IsMainEditor`로 역할 자동 분기 → 메인=Host, 클론=Client 자동 접속.
- **fallback(2nd 프로세스 직접 런치)은 불필요** — 영속 활성으로 per-run 0 이미 충족. 채택 보류(백로그).

**확정 채택 — Relay 우회 로컬 직결 (디렉터 A 승인):**
- 현 멀티 진입은 `GameNetworkManager.HostRelayAsync/JoinRelayAsync`(Relay JoinCode). 가상 플레이어는 **동일 머신** → Relay 불필요.
- transport 기본값이 이미 `127.0.0.1:7777`. QA용 `StartLocalHost()`/`StartLocalClient()` 직접 경로 추가 → **JoinCode temp 공유 제거**. Relay 경로는 실제 플레이용으로 보존, Relay 자체 검증은 P2 백로그.

### 4-1. 스냅샷 스키마 (확정 — 디렉터 C 명제별 비교축 반영)

역할별 동일 포맷. 파일: `Temp/mppm-qa/probe_host.json` / `probe_clone.json`.

```jsonc
{
  "role": "host|clone",
  "isMainEditor": true,
  "localClientId": 0,
  "connected": true,
  "connectedClientCount": 2,
  "timestamp": 0.0,
  "objects": [                 // SpawnManager.SpawnedObjects 전역 열거
    { "id": 5,                 // NetworkObjectId (양측 동일 키 — clientId 아님)
      "kind": "player|monster|dropitem|building|other",  // GetComponent 타입 판별
      "pos": { "x":0, "y":0, "z":0 },
      "hp": 80, "maxHp": 100,  // 캐릭터(player/monster)만. CharacterBase.HP/MaxHP
      "alive": true,           // monster만. !CharacterBase.IsDead (V5 디스폰 전 사망상태 정밀)
      "cold": 0, "maxCold": 100, // player만. PlayerCharacter.Cold/MaxCold (V7 P0 편입 확정)
      "itemId": 1, "count": 5  // dropitem만. WorldDropItem.ItemInfo.Id/Count (NetworkVariable Read=Everyone → 클론 읽힘). V5 드롭종류·V6 월드아이템 식별.
    }
  ],
  "localInventory": [ { "itemId": 1, "count": 5 } ]  // 본인 로컬 인벤(복제 안 됨, 단측). V6용.
}
```

**대조 규칙 (QA가 적용):**
- `objects`: `id`(NetworkObjectId) 키 매칭 → 존재 일치(V1/V3/V5) + 값 일치(`hp`/`cold` V4/V7, `pos` V2, dropitem `itemId` V5/V6).
- `localInventory`: 클론 단측 확인(V6 수집자 +1). 호스트는 클론 인벤 미보유가 정상 — 양측 대조 대상 아님.
- 종류 판별: `PlayerCharacter`/`MonsterBase`/`WorldDropItem`/`WorldBuildingObject` 컴포넌트 유무.
- 수렴시간: `mppm_collect` 반환 `{host:{…,timestamp}, clone:{…,timestamp}}` 양측 timestamp(Time.realtimeSinceStartup) 시각차로 판정.

**P1 주의(V8 건물):** `WorldBuildingObject.BuildingInfoId`는 **plain int(NetworkVariable 아님)** → 클론에서 값 보장 안 됨. building 종류값 양측 대조는 V8(P1) 구현 시 별도 동기화 검토 필요. V8 자체가 P1이라 P0 스키마엔 영향 없음.

## 5. 네트워크 동기화 항목

- **신규 NetworkVariable: 없음.** 본 기능은 관찰 전용.
- **신규 ServerRpc/ClientRpc: 없음** (시나리오 자체는 TestManager가 기존 호스트-권위 경로 사용).
- **권위 위치**: Host(메인 에디터)가 서버 권위 그대로. 클론=클라이언트.
- 직결 채택 시 `StartHost()`/`StartClient()` 호출 추가뿐 — NGO 동기화 의미론 무변경.

## 6. 상태 머신 (부트스트랩 Worker)

유효 상태: `Idle → DetectingRole → (Host: StartingHost → HostReady) / (Client: WaitingForHost → Joining → ClientReady) → Failed`

가능 전이:
- `DetectingRole → StartingHost` (IsMainEditor==true)
- `DetectingRole → WaitingForHost` (IsMainEditor==false)
- `Joining → ClientReady` (StartClient 성공 & connected)
- `* → Failed` (타임아웃/네트워크 매니저 null)

불가능(명시 차단):
- 메인 에디터가 Client로 진입 금지.
- 이미 `IsListening`인데 재시작 금지(중복 호출 가드).
- `Failed` 후 자동 무한 재시도 금지(1회 로그 후 정지).
