# 기능 개발 흐름 (레벨 수치 주도) — 작업 계획서

> **로드맵 매핑**: R1~R5 공통 적용 (특히 R3 콘텐츠 채우기). 한 기능/콘텐츠 단위마다 이 흐름으로 진행.
> **작성**: 2026-06-08 / **상태**: 🆕 (청사진 — 구체 대상은 작업 시 지정)
> **투입 멤버**: `director`(기획 제공) · `level-design`(수치) · `client`(개발) · `qa`(검증) + `designer`(④에서 추가 spawn)
> **대상 기능**: `<작업 시 지정>` (예: 특정 지역 몬스터·드롭, 특정 시스템 수치)

---

## 0. 흐름 한눈에

```
[상위] director ─ 무슨 기능·왜 (출시 기획, 이미 완료) ※ level-design은 기능 결정 안 함
          ↓
①  level-design ─ 그 기능의 수치 명세 (무엇을 얼마로) ← 작업 시작점
          ↓
②  client       ─ 수치대로 기능 개발
          ↓
③  qa           ─ 수치 합격 기준으로 검증
          ↓ (완료)
④  designer 추가 ─ 필요 리소스 생성
```

> 핵심: **level-design 수치 명세 = client 개발의 계약 + qa 검증의 정답지.** 한 문서가 입력·검증을 동시에 한다.

## 1. 목표 (Definition of Done)

대상 기능이 아래 4단계를 모두 통과하면 완료:

- [ ] **① 수치 명세 완료** — level-design이 대상 기능의 수치를 측정 가능하게 박음(QA 합격 기준 포함, [수치명세 양식](../../level-design/출시_지역별_수치명세/수치명세.md) §9 형식)
- [ ] **② 개발 완료** — client가 수치대로 기능 구현, 컴파일 0에러
- [ ] **③ 검증 PASS** — qa가 수치 합격 기준으로 검증(정적 일치 / 확률 ±10%p / 곡선 ±허용)
- [ ] **④ 리소스 적용** — designer가 필요 프리팹·자산 생성·적용

## 2. 선행 의존

- ✅ 상위 출시 기획(director) — 무슨 기능·왜는 이미 정해짐(출시 기획 문서군)
- 대상 기능이 R1(스키마·시스템) 의존이면 → R1 먼저 (신규 CSV 컬럼·시스템 토대)

## 3. 작업 분해

| T# | 태스크 | 담당(영역) | 의존 | 산출물 |
|----|--------|-----------|------|--------|
| T1 | 대상 기능 **수치 명세** 작성 | level-design | director 기획 | 수치 명세(합격 기준 포함) |
| T2 | 수치대로 **기능 개발** | client | T1 | 코드명세 + 구현(컴파일 0) |
| T3 | **수치 합격 기준 검증** | qa | T2 | QA 시나리오 + PASS 리포트 |
| T4 | 필요 **리소스 생성** | designer(추가 spawn) | T3 | 프리팹·자산 + 적용 |

> 경계: level-design은 수치(②)만, 기능 결정(①)은 director. 코드 디테일은 client 코드명세. 리소스는 designer.

## 4. 체크포인트 (사용자 확인)

- **T1 후**: 수치 확정 — 개발 입력이자 검증 기준이므로 먼저 검토.
- **T3 후**: 검증 PASS — 리소스 작업 전 기능 동작 확인.
- **T4 후**: 리소스 적용 완료 → 기능 단위 종료.

## 5. 리스크

- **수치 명세 모호 → 개발 헛돔**: DoD는 측정 가능해야(정성 표현 금지). 모호하면 T1에서 멈춤.
- **신규 컬럼 필요**: 수치가 기존 CSV 스키마 초과 시 → client에게 스키마 의뢰(R1 영역).
- **리소스 ↔ 코드 슬롯 불일치**: T4 designer가 client 슬롯명에 맞춰 인터페이스 명세 교환.

## 6. 진행 추적

- [ ] T1 수치 명세 (level-design)
- [ ] T2 기능 개발 (client)
- [ ] T3 검증 (qa)
- [ ] T4 리소스 (designer)
- [ ] 기능 단위 완료 보고
