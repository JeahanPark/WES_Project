# WES 문서 양식 가이드

> **목적**: 4종 문서(스케줄·작업 계획서·작업·에이전트)의 frontmatter·구조·위치·컨벤션을 표준화한다. 새 문서는 이 양식으로 시작.
> **원칙**: frontmatter·위치·의뢰표는 **강제**, 본문 섹션은 **권장**(유형별 표준 섹션 참조). 1인 개발 부담 방지 — 빡빡함보다 일관성.
> **갱신**: 2026-06-08 신설.

---

## 0. 공통 컨벤션 (4종 모두)

- **언어**: 한국어.
- **wiki link/상대링크**: 상위·연관 문서는 반드시 링크. 고아 문서 금지.
- **상태 이모지**: ✅완료 · 🔄진행 · ⏸️보류 · 🆕신규 · ⚠️부분/주의 · ❌미착수 · 🎯대상.
- **수치**: 측정 가능하게(특히 레벨/밸런스). 정성 표현("적당히") 지양.
- **영역 경계 명시**: 이 문서가 *무엇까지* 다루고 *무엇은 타 영역*인지 frontmatter `성격`에 1줄.

---

## 1. 스케줄 문서 양식

> 대상: `document/WES_Schedule.md` (프로젝트 일정·로드맵). 1개 유지(분산 금지).

### 구조
```
# <프로젝트> — 개발 일정
> 작성일 / 개발 형태 / 작업량 / 최종 목표

## 마일스톤 요약          ← 표: 마일스톤 | 기간 | 목표 | 상태(이모지)
> **YYYY-MM-DD 갱신**: ...  ← 시점 기록(날짜 prefix, 이력 누적·과거 노트 삭제 금지)

## <마일스톤>별 상세        ← 체크박스 [x]/[ ]
## <출시>본 로드맵          ← 의존성 순서 R1→RN, 각 R에 영역 분담(client/designer/...)
## 리스크                  ← 표: 리스크 | 대응
## 전체 타임라인           ← ASCII
```

### 규칙
- **갱신은 추가**: 과거 시점 노트(날짜 prefix)는 지우지 말고 새 노트 append → 의사결정 이력 보존.
- **공수(시간) 추정은 client 영역** — director/스케줄은 구조·순서·우선순위·의존성까지.
- 로드맵은 **의존성 순서**(R1 토대 → … → 출시). 병렬 가능 항목 명시.

---

## 2. 작업 계획서 (팀 작업 단위)

> 대상: 로드맵 한 단위(R1 등)나 한 주제를 팀이 실행할 때의 **단위 실행 계획**. 스케줄(전략)과 작업 문서(산출) 사이의 전술 레이어. team-lead(메인 세션)가 작성. 완료 시 `_archive`로.
> 위치: `design/work-plans/<주제>/계획.md`.

### 구조
```
# <작업명> — 작업 계획서
> **로드맵 매핑**: R1 (스케줄 어느 단위인가)
> **작성**: YYYY-MM-DD / **상태**: 🆕/🔄/✅
> **투입 멤버**: <선택 spawn한 에이전트와 이유>

## 1. 목표 (Definition of Done)   ← "끝났다"의 명확한 기준(측정 가능)
## 2. 선행 의존                   ← 이 작업 전에 끝나야 할 것
## 3. 작업 분해                   ← 표: T# | 태스크 | 담당(영역) | 의존 | 산출물
## 4. 체크포인트                  ← 사용자 확인 지점
## 5. 리스크
## 6. 진행 추적                   ← 체크박스 [x]/[ ]
```

### 규칙
- 로드맵 R단위 ↔ 작업 계획서 1:1 또는 1:N.
- **DoD 명확히** — 모호하면 시작 금지(측정 가능한 완료 기준).
- 담당은 7인 풀에서 **선택 spawn**(TEAM_PROCESS §0-1 분기 먼저).
- 계획서는 *무엇을·누가·순서*까지. 코드 디테일(클래스명·시그니처)은 client 코드명세 영역 — 추측 박지 말 것.
- 완료 시 `design/work-plans/_archive/`로 이동.

## 3. 작업 문서 양식

> 대상: 기획·명세·서사·사운드 등 실제 산출물. 위치는 유형별 폴더(아래).

### 2.1 공통 frontmatter (강제)
```
# <주제> — <유형>
> **버전**: vX.Y (<작성 렌즈/에이전트>)
> **작성**: YYYY-MM-DD (<작성자>)
> **상위**: [상위 비전·캐논 문서](경로)   ← 무엇에서 파생됐는가
> **성격**: 이 문서가 다루는 영역 경계 (무엇까지 / 무엇은 타 영역으로 위임)
> **연관**: [관련 문서] (선택)
```
- 개정 문서는 `> v0.1→v1.0 변경: ...` 한 줄 추가.

### 2.2 유형별 위치 · 표준 섹션

| 유형 | 위치 | 표준 섹션 | 소유 에이전트 |
|------|------|-----------|---------------|
| 기획서 | `design/game-design/<주제>/기획.md` | 한줄컨셉·디자인골·**안티골**·경험흐름·핵심결정·콘텐츠분량·톤·밸런스·엣지케이스·향후확장·참고 | director |
| 코드명세 | `design/client-spec/<주제>/코드명세.md` | 영향범위·재활용·신규클래스·데이터레이어·네트워크·상태머신·이벤트계약·의존성·구현단계·비용·엣지케이스·권고 | client |
| 레벨명세 | `design/level-design/<주제>/` | 지역개요·페이싱·스폰·드롭·배치·의뢰·**QA합격기준**(측정법·허용오차) | level-design |
| 서사 | `design/narrative/<주제>/` (서사.md·텍스트.md) | 집필대상·텍스트(화자태그·placeholder)·placeholder표·반영위치·캐논메모 | story |
| 사운드 | `design/sound/<주제>/사운드명세.md` | 오디오이벤트·SFX스펙·BGM·variation·믹싱·파일매핑·백로그 | sound |

> 기획↔코드명세는 `<주제>/` 이름을 **맞춰** 쌍을 이룬다(game-design ↔ client-spec).

### 2.3 공통 말미 (권장)
- **의뢰 표**: 영역 분기 — `client(스키마·코드) / designer(자산) / level-design(수치) / sound(음원) / story(텍스트)`.
- **임시 → 출시 변경점** 표 (개정 문서).
- **참고 / 상위 링크**.

## 4. 에이전트 문서 양식

> 대상: `.claude/agents/*.md` (서브에이전트 정의서). 현재 7인: director·client·designer·qa·sound·story·level-design.

### 3.1 frontmatter (강제)
```
---
name: <소문자-케밥>
description: <언제·무엇을 하는지 트리거 서술. 영역 경계 포함>
tools: <PoLP — 역할에 필요한 최소 도구만>
model: opus
---
```
- **PoLP**: 도구는 최소 나열. 문서형(director/story)은 MCP 없이, 자산형(designer/sound)은 해당 MCP만.

### 3.2 필수 섹션
```
## 정체성과 사고 영역        ← 무엇만 사고/발화하는가
## 절대 사고하지 않는 영역    ← 타 영역 경계(금지)
## 파일 접근 규칙            ← 읽기 / 쓰기 / 절대 안 됨
## 워크플로우               ← 단계별 절차
## <표준 산출물 섹션>        ← 해당 시(명세 표준 섹션 등)
## 인터페이스               ← 표: 방향 | 발신→수신 | 내용
## 토론 자세
## 팀 운영 절차             ← 7인 풀·선택적 spawn 명시
## 팀 메일박스 처리 (팀 모드 전용)  ← 공통 블록(아래)
```

### 3.3 공통 블록 — "팀 메일박스 처리" (전 에이전트 동일 복사)
inbox 최우선 / "첫 턴 제한" 무시 / idle은 inbox 빈 때만 / 후속 턴 답변 시 명시 보고. (기존 정의서들의 동일 문단 사용.)

### 3.4 경계 철칙
- 각 에이전트는 자기 영역만. 타 영역 필요 시 **SendMessage 의뢰**(직접 침범 금지).
- 직접 편집 권한이 있는 경우(level-design CSV/씬)는 **충돌 방지 철칙** 섹션 필수.
